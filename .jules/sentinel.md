## 2024-03-14 - Explicit Security Boundaries Over Framework Defaults
**Vulnerability:** System configurations (like SSH daemon and Firewall) relied on upstream OS/framework defaults, which leaves the system vulnerable to brute-force attacks and network intrusions if upstream defaults change or are not secure enough by default (e.g. implicitly allowing SSH password auth).
**Learning:** A core security pattern for this repository is to always explicitly define security boundaries and mechanisms (like `networking.firewall.enable = true;` and `services.openssh.settings.PasswordAuthentication = false;`) rather than relying on framework or OS defaults, ensuring intended security posture even if upstream defaults change.
**Prevention:** Always explicitly define critical security settings. For SSH, explicitly configure `PasswordAuthentication = false;` in `services.openssh.settings`. For networking, explicitly define `networking.firewall.enable = true;`. Use `settings` attribute sets for modern NixOS service configurations rather than deprecated top-level options.
